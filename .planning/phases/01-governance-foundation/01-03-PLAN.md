Create governance/rfcs/README.md describing the RFC process:

**RFC Process**

1. **What is an RFC?**
   - Request for Comments - formal proposal for specification changes
   - Required for: New features, breaking changes, significant modifications
   - Not required for: Typos, clarifications, editorial fixes

2. **Who can submit?**
   - Anyone (Open Contribution per CONTEXT.md)
   - No membership required to submit
   - TSC reviews based on merit, not submitter status

3. **RFC Lifecycle**
   - Draft: Author prepares RFC using template
   - Submitted: PR opened against rfcs/ directory
   - Champion assigned: TSC member shepherds the RFC (prevents RFC abandonment per research pitfall)
   - Review period: Public comment period based on change type
   - Decision: TSC votes (Accept/Reject/Defer)
   - Implementation: If accepted, merged and implemented in target version

4. **Review Periods** (from CONTEXT.md decisions)
   - Minor changes (non-breaking): 2 weeks
   - Major changes (new features): 30 days
   - Breaking changes: 60 days (plus veto period)

5. **Decision Process**
   - Lazy consensus: If no objections by deadline, RFC proceeds
   - Explicit vote: For contested RFCs, TSC votes
   - Breaking changes: Require veto-free TSC approval (see CHARTER.md)

6. **RFC Numbering**
   - Sequential integers: 0001, 0002, 0003...
   - Assigned when RFC is accepted (not at submission)
   - Draft RFCs use placeholder: XXXX

7. **Language**
   - Official language: English (per CONTEXT.md)
   - Community translations encouraged but not authoritative

8. **RFC Statuses**
   - Draft: Work in progress
   - Submitted: Under review
   - Accepted: Approved for implementation
   - Implemented: Released in specification version
   - Rejected: Not accepted (with rationale documented)
   - Withdrawn: Author withdrew
   - Deferred: Postponed for future consideration

Include a table of active RFCs (initially empty) and link to 0000-template.md.

Structure with 9 sections:

```markdown
# RFC-XXXX: [Title]

- **RFC Number:** XXXX (assigned on acceptance)
- **Title:** [Descriptive title]
- **Author:** [Name (organization)]
- **Champion:** [TSC member - assigned after submission]
- **Status:** Draft | Submitted | Accepted | Implemented | Rejected | Withdrawn | Deferred
- **Created:** [YYYY-MM-DD]
- **Review Deadline:** [YYYY-MM-DD] (calculated from submission based on change type)
- **Spec Version Target:** [e.g., 1.3.0]

## Summary

One paragraph explanation of the proposal.

## Motivation

Why is this change needed? What problem does it solve?
- What use cases does this enable?
- What pain points does this address?
- Who benefits from this change?

## Guide-level explanation

Explain the proposal as if teaching it to an adopter who will implement it.
- Use examples and scenarios
- Explain concepts in plain language
- Focus on the "what" and "how to use"

## Reference-level explanation

Technical specification of the proposal.
- Precise definitions and semantics
- Edge cases and corner cases
- Interaction with existing features
- Schema changes, if applicable (JSON-LD, etc.)

## Drawbacks

Why should we NOT do this?
- What are the costs?
- What complexity does this add?
- Who is negatively impacted?

## Rationale and alternatives

Why is this the best design among alternatives?
- What alternatives were considered?
- Why were they rejected?
- What is the impact of not doing this?

## Prior art

How do other standards/systems handle this?
- Existing approaches in luxury industry
- Approaches in adjacent domains (supply chain, identity, etc.)
- Lessons learned from prior art

## Unresolved questions

What questions need resolution during the RFC process?
- Implementation details to be determined
- Open design questions
- Dependencies on other work

## Future possibilities

What extensions or follow-on work might this enable?
- Not a commitment, but potential directions
- How might this evolve?

- **Date:** [YYYY-MM-DD]

```
